## What StarNix is
A Nutanix **NCP-MCI** certification study game: a shell that boots a shared cinematic + a menu of three sub-games that all teach the same exam material via spaced retrieval. Enemy: the **BCM (Broad Communication Military)**. Crew: the **Nutanix Starlight Rescue Crew (NX-SRC)**, a.k.a. the **StarNix Rescue Crew**.
- **ARM** — Acropolis Rescue Mission (2D flight + collect): rebuild the MCI station.
- **KBB** — Kuiper Belt Battle (2D roguelike, Slay-the-Spire × Balatro): hunt the BCM warship.
- **CC** — Chasm Chase (3D Temple-Run, Three.js): chase the escaped squadron down a canyon.

## Build model (now)
Plain-JS modules, **no build step**, each attaches to `window.StarNix`. The Core chat **concatenates** them into one `index.html` for Google Apps Script. (TypeScript/Vite/clasp deferred to containerization.) Three.js r128 loaded from cdnjs in `<head>` for CC.

**Commands** (run in the folder with the module files):
- `node build.mjs` → regenerates `index.html` (order: core → **assets** → shell → audio → arm → cc → kbb → boot).
- `node verify-build.mjs` → headless jsdom integration check; must print `ALL GREEN`.
- `node audio-smoke.mjs` → mock-AudioContext check of the synth path (jsdom has no AudioContext); must print `AUDIO SMOKE: ALL GREEN`.
- `node scheduler-test.mjs` → unit-tests the Leitner scheduler; must print `SCHEDULER TEST: ALL GREEN`.
- `node multi-answer-test.mjs` → multi-select schema/grading; must print `MULTI-ANSWER TEST: ALL GREEN`.
- `node shuffle-test.mjs` → answer-order shuffle (learning integrity); must print `SHUFFLE TEST: ALL GREEN`.
- `node timer-test.mjs` → dynamic per-question timer formula; must print `TIMER TEST: ALL GREEN`.
- `node import-questions.mjs` → authoring `.md` → `questions.js`; must print `VALIDATION: ALL GREEN`.
- **`audio-test.html`** → open in a browser to actually *hear* the tracks/SFX (the only real audio check). Needs `audio.js` beside it.
- `node balance-probe.mjs` → headless KBB balance probe (auto-plays runs at various accuracies, reports depth reached / min HP / coins / death cause). For tuning, not a pass/fail gate.

**Deploy to Apps Script:** project has `Code.gs` with `doGet(){ return HtmlService.createHtmlOutputFromFile('index'); }` + an HTML file named `index` containing `index.html`. Update = paste new `index.html` → Deploy → Manage deployments → new version.

## The frozen contract (01 §9a) — brief
- `window.StarNix = { core, shell, registerGame(m), getGame(id), registerAudio(impl), boot(root,opts), initCore(opts), makeContext(gameId) }`.
- A game = `{ id:"ARM"|"KBB"|"CC", mount(root, ctx), unmount() }`; `unmount` must leave **zero residue** (cancel RAF, remove listeners, free DOM).
- `ctx` (per game): **guaranteed** `questions, mastery, persistence, rng, audio, theme, telemetry`; **additive** `ai, settings, sanitize`; and `exit()` (added by the shell's `enterGame` so a game can request return-to-menu).
- Audio engine installs at `window.StarNix.core.audio` (audio.js). 5 tracks: `cinematic, menu, arm, kbb, cc`. The shell plays the right track on enter; each game also calls `playTrack` itself.
- Verification is **structural** (wired, no errors across frames, clean lifecycle). It does NOT prove visual correctness, that CC's WebGL renders, or that audio sounds right — those are real-browser-only.

### CORE/audio — fix music degradation over time (scheduler catch-up clamp) — 2026-06-24 — ✅ build green (195/195) · v0.6.16
- **Bug (Jason playtest):** music "starts to go slow and then sounds really weird after a bit of time." Root cause in `audio.js` `TrackPlayer._schedule`: it's a look-ahead scheduler (`setInterval` every 25 ms schedules notes up to `currentTime + 0.12`). `AC.currentTime` advances in real time, but the `setInterval` is **throttled when the tab is backgrounded** and **delayed by GC / heavy CC WebGL frames**. After a stall, `nextNoteTime` is far behind `currentTime`, and the `while` loop replayed **every missed 16th at its original (now-past) timestamp** — Web Audio fires past-dated source nodes immediately, so you get a burst of overlapping notes **plus** a wall of oscillator/gain nodes allocated in one synchronous tick. Repeated stalls compound it → the audible "slow then garbled."
- **Fix:** catch-up clamp at the top of `_schedule` — if `nextNoteTime < currentTime`, fast-forward silently to the present (snap on a >1 s gap; otherwise skip the missed steps while preserving pattern phase) instead of sounding them. Self-contained; public API and the documented two-clocks behaviour unchanged.
- **Verification:** `audio-smoke.mjs` now drives the mock clock through a **30 s gap** and asserts one post-gap tick schedules a **bounded** number of notes (oscillator-allocation proxy, `< 64` vs ~900 unclamped) and that the next tick is back to steady state. **Negative control:** disabling the clamp fails the bounded-burst assertion (`AUDIO SMOKE: 1 FAIL`). Integration still 195/195; all 5 harnesses green.
- Files: `audio.js`, `audio-smoke.mjs`, `starnix-core.js` (version) (+ `index.html`), `STATE.md`, `CHANGELOG.md`. **Spec `08` is Audio-chat-owned — flag for them to note the resync behaviour; no design change.**
