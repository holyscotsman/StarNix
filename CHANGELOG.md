History of changes, reverse-chronological. **Append-only — never rewritten.** The spec files (`00`–`05`) are the source of truth for *current intended design*; this file is the source of truth for *what happened*. Every code change adds one entry. Format follows Keep-a-Changelog.

Sections per entry: **Added · Changed · Fixed · Removed**. Each line: `<what> — <why>. Files: <paths>. Tests: <result>.`

## [0.139.0]
- **V1.1 CC#1 — the boost is EARNED: answer-charged boost meter (Chasm Chase).** The old boost fired on a flat gate-count cadence (`_gatesPassed % GATES_PER_BOOST`), so the run's biggest reward arrived whether you knew the material or not. Now `CCSim` carries a `boostCharge` meter (0..`GATES_PER_BOOST`): each CORRECT gate answer adds a charge and a full meter arms the boost (still fired on resume, exactly as before); a WRONG answer or timeout drains half the charge (`floor(charge/2)`); `_gatesPassed` keeps counting as a stat but no longer arms anything. The HUD gained a small gold "Boost" charge bar beside the shield pips (fills per correct, `hot` glow at one-short-of-full, iris/aqua `riding` fill while the boost is live; cached — only repaints when the charge changes). The G2 checkpoint carries `boostCharge` so a resumed run keeps its earned charge. Pins: cc-run gained a dedicated charge-flow group (empty-meter miss no-underflow / half-charge-buys-nothing / second-correct-arms / fires-on-resume / miss-drains-half / gate-count-alone-never-arms, 76 total) and the two old cadence drives now seed `boostCharge` instead of `_gatesPassed`; verify-build's boost block rewritten to the charge model (drain + arm + fire) with mastery snapshot/restore so the sim-level answers stay state-neutral for the downstream due-queue pins, and the CC resume pin now restores `boostCharge`. Negative control: disabling the charging in `answer()` failed exactly the 6 cc-run + 3 verify-build charge/boost pins; restored. Gate: 18 suites ALL GREEN.

## [0.131.0] - 2026-07-04

## [0.130.0] - 2026-07-04

- **V1.1 Backend#1 — save-file safety net.** Weeks of Leitner history lived in one localStorage key behind a 400ms debounce with zero protection: no flush on tab close (answers inside the debounce window silently lost), corrupt JSON meant a silently FRESH profile ("total loss by design"), and there was no way to move progress between devices. Now: (1) **auto-flush** — the provider flushes on `pagehide` and on `visibilitychange→hidden`, closing the debounce data-loss window (guarded, headless-safe, `autoFlush:false` opt-out for tests); (2) **last-known-good backup** — every write first rotates the previous stored state into `starnix:profile:bak`, and `load()` falls back to it when the main key is corrupt/missing before ever defaulting to a fresh profile; (3) **export/import** — new `persistence.exportProfile()/importProfile(json)` seams (import validates + migrates BEFORE touching storage, so a bad paste can't wipe anything) surfaced in Settings → Data as Export/Import with a JSON textarea; import swaps the live profile in place so menu/XP/games see it immediately. Pins: verify-build +5 (corrupt-main→backup fallback restores the older xp; export→import round-trips into a fresh store; garbage import throws and leaves storage untouched; lifecycle flush wired (source); Settings offers Export+Import+Reset). Controls: backup rotation removed → fallback pin fell; pagehide listener removed → its pin fell; both restored, gate 0. Files: starnix-core.js, starnix-shell.js, verify-build.mjs. Tests: gate exit 0.
